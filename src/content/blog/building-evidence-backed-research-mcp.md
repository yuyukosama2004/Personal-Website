给 Agent 一个搜索工具很容易，让它产出可以复核的研究结果却困难得多。搜索只回答“可能去哪里看”，研究还要回答“依据是什么、信息是否冲突、过几天还能不能复现”。

GroundedSeek 的设计起点，是把证据当作系统里的核心数据，而不是回答末尾的装饰。

## 从链接列表到证据单元

一个 URL 不足以支撑结论。页面可能更新，摘要可能断章取义，同一页面也可能同时包含支持与反对某个判断的信息。

因此系统保存的证据单元包含：

```json
{
  "sourceUrl": "https://example.com/document",
  "title": "Document title",
  "retrievedAt": "2026-07-14T10:00:00Z",
  "excerpt": "The relevant passage...",
  "claimIds": ["claim-12"],
  "contentHash": "sha256:..."
}
```

`retrievedAt` 说明观察时间，`contentHash` 帮助判断来源是否变化，`claimIds` 则让证据与具体结论建立关系。最终答案中的每个重要判断都可以向下追溯。

## MCP 工具应该小而明确

把一个通用浏览器直接交给 Agent，会产生难以控制的操作空间。GroundedSeek 的 MCP 接口更接近研究动作：创建项目、搜索、采集来源、登记证据、生成综合报告、导出 artifact。

例如，登记证据的输入可以被约束为：

```ts
type AddEvidenceInput = {
  projectId: string;
  sourceUrl: string;
  excerpt: string;
  supports: string[];
  contradicts?: string[];
};
```

这比一个接受任意脚本的工具更容易校验，也更容易在日志中理解 Agent 做了什么。

## 搜索、浏览与综合需要分层

研究流水线至少包含三个不同阶段：

1. **发现**：用多组查询覆盖主题，形成候选来源集合。
2. **验证**：打开原始来源，记录发布日期、作者、上下文和关键片段。
3. **综合**：比较证据，标记一致、冲突与未知，而后形成结论。

如果搜索摘要直接进入综合阶段，系统会把搜索引擎的截断文本当成原始证据。分层意味着多一些状态，但能显著减少这种错误。

## 冲突不是异常

真实研究经常遇到来源不一致。系统不应该为了生成流畅答案而自动消除冲突，而要保留双方证据，并说明差异可能来自统计口径、时间或利益关系。

在数据模型中，结论可以拥有状态：

- `supported`：存在充分且相互独立的支持；
- `contested`：可靠来源给出相互冲突的信息；
- `insufficient`：信息不足，暂时不能下结论。

“不知道”是研究系统必须能够稳定表达的结果。

## 版本化 artifact 让研究可继续

一次研究往往不是终点。新版本发布、法规变化或新的原始资料都可能改变结论。GroundedSeek 将查询、来源、证据和报告作为一个可版本化项目保存，并支持 Markdown 与 JSON 导出。

Markdown 适合人阅读和进入知识库，JSON 适合后续程序处理。两者来自同一份结构化状态，避免分别维护后产生偏差。

## Local-first 的实际意义

本地优先不仅是隐私偏好，也让研究资料的生命周期不依赖某个在线聊天窗口。使用者可以备份数据库、用自己的工具搜索导出文件，并决定何时删除敏感项目。

这并不意味着拒绝网络服务，而是让网络承担获取信息的角色，让研究资产的控制权留在本地。

## 仍然需要确定性验证

Agent 可以归纳证据，但外部代码应检查一些硬约束：引用的证据是否存在、URL 是否有效、报告中的 claim 是否都有来源、导出是否符合 schema。这些检查不判断观点，却能阻止结构性错误悄悄进入成品。

证据驱动的 Research MCP 本质上是一套状态机和数据契约。浏览能力当然重要，但只有把来源、结论和版本连起来，模型生成的研究才真正具备被复核、被更新和被信任的基础。
